# SEO & GEO SOP

Checklist for maintaining the search-engine and generative-engine optimization work on this blog as new posts and pages are added.

**How to use:** when you finish writing a post, open the "Every new post" section and tick through it before merging. For new `_tabs/` pages, use the "Every new page" section. Do the "Periodic checks" on a cadence (monthly).

---

## 0. Foundations (already in place — don't redo unless broken)

These are set site-wide and should rarely change. Knowing what's there is enough.

| Concern | Where it lives | Why |
|---|---|---|
| Google Search Console verification | `_config.yml: webmaster_verifications.google` | Ownership for GSC, emits meta tag |
| Bing Webmaster verification | Imported from GSC (Bing dashboard) | Bing result feeds ChatGPT Search, DuckDuckGo |
| Site description / tagline | `_config.yml: description, tagline` | Renders into homepage og:description |
| Site-wide author JSON-LD | jekyll-seo-tag auto (from `social.name`) | Basic WebSite schema |
| Person JSON-LD (author hub) | `_tabs/about.md` (inline `<script>`) | Ties the author identity to VeloDB / Apache Doris / GitHub / LinkedIn |
| AI crawler policy | `/robots.txt` | Explicit allow for GPTBot / ClaudeBot / PerplexityBot / Google-Extended / Applebot-Extended / CCBot |
| LLM site summary | `/llms.txt` | Per [llmstxt.org](https://llmstxt.org/) — short bio + per-post summaries |
| Sitemap | Auto-generated by jekyll-sitemap at `/sitemap.xml` | Submitted to GSC and Bing |
| OG default image | `_config.yml: social_preview_image` | Fallback for pages without per-post image (⚠️ **not yet set** — pending Task #3) |

**Sanity check anytime:**
```bash
curl -sI https://morningman.github.io/sitemap.xml | head -1   # expect 200
curl -sI https://morningman.github.io/llms.txt | head -1      # expect 200
curl -sI https://morningman.github.io/robots.txt | head -1    # expect 200
```

## 1. Every new post — pre-publish checklist

### 1.1 Front matter

Required fields (all present, in this order):

- [ ] `title` — specific, contains the main concept/entity, under 70 chars
- [ ] `date` — with timezone offset (e.g. `2026-05-01 10:00:00 -0700`)
- [ ] `categories` — 1–2 from existing vocabulary (e.g. `[Data Engineering, Apache Iceberg]`)
- [ ] `tags` — **4–7 tags** from the fixed vocabulary in §3 below. Do NOT invent new one-off tags
- [ ] `description` — 120–160 chars, contains main keyword, reads as a standalone search snippet
- [ ] `image.path` — absolute path to OG image at 1200×630 (see §1.3)
- [ ] `image.alt` — accessible description of the OG image

Example:
```yaml
---
title: "Apache Paimon on Doris: Streaming Meets MPP"
date: 2026-05-15 10:00:00 -0700
categories: [Data Engineering, Apache Paimon]
tags: [apache paimon, apache doris, lakehouse, cdc, streaming]
description: "How Apache Doris integrates with Paimon's LSM-on-object-storage design, and what changes when streaming writes meet MPP analytics."
image:
  path: /assets/img/posts/2026-05-15-paimon-doris-og.png
  alt: "Apache Doris reading from a Paimon table with streaming writes flowing in"
---
```

### 1.2 Content structure

- [ ] **Opening paragraph** — sets up the scenario/problem concretely in 2–3 sentences, avoids AI-slop framing ("in today's fast-paced world...")
- [ ] **TL;DR block** — 4–7 bullets after the intro, before the first H2. This is what skim readers and LLM citations pick up. See Iceberg V3 post for the pattern
- [ ] **Heading hierarchy** — H2 for major sections, H3 for sub-sections. Chirpy auto-generates TOC from these
- [ ] **Named entities in H2s** — product names (Apache Doris, Iceberg V3, Variant), not just topic words. Helps LLM retrieval
- [ ] **Internal links** — link to at least 1–2 related existing posts where the topic genuinely connects. Creates the topic cluster signal
- [ ] **External links** — link to primary sources (specs, docs, PRs, papers), not SEO-chasing roundup articles
- [ ] **English only** — no Chinese content; the blog is English-language per editorial choice

### 1.3 Images

- [ ] **OG image generated** using the whiteboard-sketch brand style. See `prompt/` directory for the prompt template and per-post prompts
- [ ] **Saved at** `/assets/img/posts/<YYYY-MM-DD>-<slug>-og.png` — 1200 × 630, under 300 KB
- [ ] **Inline diagrams** where useful — follow the same whiteboard style for visual consistency
- [ ] **Alt text** on every image (including inline ones)

### 1.4 Pre-publish local verification

- [ ] `bundle exec jekyll serve` locally, navigate to the post, confirm:
  - TOC renders and is accurate
  - OG image preview looks right (use browser devtools → Elements → search `og:image`)
  - All internal links resolve
  - Code blocks render correctly
- [ ] View source and confirm `<meta name="description">`, `<meta property="og:image">`, and the BlogPosting JSON-LD all contain sensible values
- [ ] If adding new terms/tags/named entities, make sure the tag vocabulary in §3 still holds; update if genuinely warranted

## 2. Every new post — post-publish checklist

After `git push` and Pages deploy completes:

- [ ] **CI passes** — GitHub Actions run on the commit is green (html-proofer validates internal links)
- [ ] **Update `llms.txt`** — add an entry for the new post under `## Posts`. Write 2–3 sentences that stand on their own as a summary (not just a copy of `description`)
- [ ] **Google Search Console: Request indexing**
  - URL Inspection → paste new post URL → Request indexing
  - Daily quota ~10–20 URLs; sufficient for a weekly publishing cadence
- [ ] **Bing Webmaster: URL Submission**
  - URL Submission → submit new URL
  - Quota is 10,000 URLs/day, very generous
- [ ] **Cross-link from existing posts if appropriate** — if the new post fills a gap that older posts hinted at, go back and add a link from the older post

## 3. Canonical tag vocabulary (15 tags)

**Never invent new one-off tags.** Reuse from this list. If a genuinely new topic area emerges that applies to 2+ current/planned posts, add it here first.

| Category | Tags |
|---|---|
| Engines / products (3) | `apache doris`, `apache iceberg`, `apache parquet` |
| Topics (7) | `lakehouse`, `open format`, `semi-structured`, `indexing`, `cdc`, `query engine`, `ai` |
| Retrieval (3) | `full text search`, `vector retrieval`, `hybrid search` |
| Format / spec (2) | `iceberg v3`, `variant` |

**Format rules:** lowercase, spaces between words (jekyll-archives slugifies for URLs automatically).

**Rejected (for reference, don't re-add):**
- Too broad: `big data`, `architecture`, `analytics`
- Too narrow / one-off: `deletion vector`, `row lineage`, `merge into`, `puffin`, `subcolumn extraction`, `json`, `index`, `dml`
- Vendor-specific non-Apache: `snowflake` (let body prose handle vendor comparisons)

## 4. Every new `_tabs/` page

Rare — for standalone navigation pages (About, Games, Lakehouse pillar, etc.).

- [ ] Front matter: `icon` (FontAwesome class), `order` (integer, decides nav position), `title` (if different from filename)
- [ ] `description` field in front matter — otherwise jekyll-seo-tag falls back to extracting first body paragraph, which is usually low-value. See `_tabs/about.md` for example
- [ ] Decide if the page warrants its own JSON-LD (Person, CollectionPage, etc.) — usually not
- [ ] Add to `llms.txt` under a relevant section if it's a substantive content hub
- [ ] Check the permalink: default is `/:title/` per `_config.yml`

## 5. Periodic checks

### Monthly

- [ ] **Google Search Console → Performance**: check last-28-days impressions, clicks, top queries, top pages. Look for pages with impressions but <1% CTR — they need better titles/descriptions
- [ ] **Bing Webmaster → Search Performance**: same drill, different audience (Bing feeds ChatGPT Search)
- [ ] **GSC → Pages → Not indexed**: any unexpected entries? Usually "Alternate page with proper canonical" is fine. "Discovered - currently not indexed" on a real post is a problem
- [ ] **Tag page audit**: visit `/tags/` — any tag with 0 posts? Remove from vocabulary if no upcoming post will use it

### Quarterly

- [ ] **External link audit**: run html-proofer with `--disable-external` removed (locally, not CI):
  ```bash
  bundle exec htmlproofer _site --ignore-urls "/^https:\/\/linkedin\.com/,/^https:\/\/www\.linkedin\.com/"
  ```
  LinkedIn returns 999/405 as anti-bot behavior — ignore
- [ ] **llms.txt freshness**: confirm every post on the site has an entry; no dead links
- [ ] **Reconsider pillar page (Task #10)**: once the lakehouse cluster reaches ≥5 posts, revisit `_tabs/lakehouse.md` as a curated hub

### As the blog grows

- [ ] When a new named topic (e.g. `streaming`, `query optimization`) has 2+ posts, consider adding it to the tag vocabulary in §3
- [ ] When posts split across clearly distinct topic clusters (lakehouse vs. AI retrieval vs. ops), start separate pillar pages
- [ ] When homepage post list gets too long (>15), consider adding pagination or featured-posts logic on `index.html`

## 6. llms.txt maintenance rule

This file is easy to forget. **Every new post must get an entry.** The entry lives under `## Posts` in chronological order (newest first). Format:

```markdown
- [Post Title](https://morningman.github.io/posts/<slug>/): 2–3 sentence summary that stands alone. Lead with the concrete finding or technique, not background.
```

Write the summary fresh — don't reuse `description` verbatim. `description` is for SERP snippets (120–160 chars, keyword-focused); llms.txt summaries can be longer (250–400 chars) and more content-specific. They're for LLM agents who want to decide whether to fetch the full post.
