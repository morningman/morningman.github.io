Apache Doris 4.1 on Iceberg V3

Running the full lakehouse lifecycle — read, write, manage, maintain — from one SQL engine.

A quick grounding before we start. Apache Doris is an open-source real-time analytics and search database built for the AI era, delivering high-concurrency, low-latency analytics and search over large datasets. Apache Iceberg is an open table format that turns a pile of Parquet files in object storage into a transactional, schema-aware table that many engines can share.

Doris has long served as a compute engine that reads Iceberg data directly, bringing low-latency analytics to the lakehouse. In 4.1, it goes further and adds full-lifecycle management on top of that read path. This post is about what changes when Doris can do more than read.

TL;DR.
  • Doris 4.1 adds full DML on Iceberg — UPDATE, DELETE, MERGE INTO — from the same SQL client you already use.
  • It lands Iceberg V3, so DML stays cheap and observable instead of accumulating debt.
  • Deletion Vectors replace position-delete files with a single Puffin bitmap per data file.
  • Row Lineage adds _row_id and _last_updated_sequence_number system columns that give CDC a real watermark.
  • MERGE INTO and Row Lineage ship as experimental — POC before production.

It's 3 AM and You Just Want to Compact Some Files

A data engineer is in Doris, investigating odd values in an Iceberg table. Queries are fast. The problem is clear: a batch of rows under one dimension is wrong and needs fixing.1

So they leave Doris. They write a Spark job, adjust the scheduler, open a PR, wait for review. More commonly, they hand the ticket to the data lake platform team next door, because they don't have cluster access. The next morning they come back to Doris to verify. The full loop took fourteen hours. The code was one UPDATE.

Iceberg made the table a neutral protocol. Interacting with that table is anything but neutral — queries live in one engine, writes in another, maintenance in a third.

The more everyday version of this pain is about small files. A CDC stream keeps writing into an Iceberg table, delete files accumulate, queries slow down. They know a rewrite_data_files run would fix it. But that procedure isn't in their hands. The platform team schedules it, reviews it, waits for a maintenance window. A task that should be one SQL statement becomes a Jira ticket.

UPDATE iceberg_tbl
   SET name = 'Alice-fixed'
 WHERE id = 1;

Apache Doris 4.1 cuts both taxes in two ways. First, it fills out Iceberg's DML so query and fix closes the loop inside one SQL client. Second, it lands Iceberg V3 so the loop can hold up architecturally over time.

1 The canonical example in every internal post-mortem I've read from a lakehouse team: a data scientist spots a wrong value during query review, and the fix is scheduled into next week's maintenance window because of who has cluster access.
